# Web Based Deployment Analysis

Attempts to deploy fine-tuned Gemma 4 models on iOS via web technologies, such as in a Progressive Web App (PWA) or directly in the browser, are currently blocked by fundamental limitations in Apple's WebKit engine, which powers Safari and all other browsers on iOS. The research confirms that no public success stories exist for this deployment path as of May 2026.

The primary and most significant technical blocker is WebKit's lack of support for the **WebAssembly (WASM) Memory64 proposal**. Modern large language models, even when quantized, require a memory address space larger than the 4GB limit imposed by the 32-bit memory model (WASM32). WASM Memory64 extends this address space, making it possible to load and run multi-gigabyte models like Gemma 4 in the browser. While this feature is available in V8 (Chrome) and SpiderMonkey (Firefox) on desktop and Android, Apple has not yet shipped it in WebKit.

This limitation directly impacts several web-based LLM runtimes:

*   **`wllama`**: As noted in the user's context, this popular web-based `llama.cpp` port explicitly requires WASM Memory64 and therefore does not work on iOS.
*   **`@mlc-ai/web-llm`**: While this project has made significant strides in web-based AI, attempts to run larger models like Gemma 4 on iOS have been stymied by the same platform constraints. A relevant MLC web-llm pull request (#3485) was noted to work in desktop browsers but not on iOS.

In conclusion, any strategy relying on running a fine-tuned Gemma 4 model within the browser on an iPhone is non-viable until Apple implements WASM Memory64 support in WebKit. This is a hard platform-level dependency that prevents the entire category of web-based deployment from succeeding on iOS.

# Summary Of Technical Blockers

## Blocker Category

Runtime Integration

## Description

A critical and frequently encountered technical blocker is the Application Binary Interface (ABI) and C Application Programming Interface (C API) mismatch between React Native wrappers and the underlying native LLM engines they depend on. This issue is exemplified by the `react-native-litert-lm` wrapper. The wrapper's native module (e.g., `HybridLiteRTLM.cpp`) is compiled against a specific version of the `LiteRT-LM` C API, as defined in its header files (e.g., `c/engine.h`). However, the underlying native engine is developed independently and evolves rapidly. When a new version of the `LiteRT-LM` xcframework is released, its C API often changes. For instance, public reports indicate that the API has seen changes in sampler enum names, the refactoring of single-constructor configuration into separate create-and-set functions, and the introduction of new tokenization APIs. When a developer attempts to use a new, custom-built model that requires the latest native engine, they might replace the old xcframework with the new one. This action breaks the wrapper because the pre-compiled native bridge code is now calling functions and using data structures that have been altered or no longer exist in the new framework, leading to an ABI mismatch and causing the app to crash or fail during engine initialization. The only solution is to fork the React Native wrapper and manually port its entire native C++ bridge code to align with the new C API, a complex and time-consuming task that negates the benefit of using a pre-built wrapper.


# React Native Specific Challenges

Deploying large language models like a fine-tuned Gemma 4 within a React Native environment presents a unique and significant set of challenges that are not as prevalent in native Swift development. These challenges primarily revolve around the abstraction layer that React Native introduces between the JavaScript application logic and the underlying native platform.

1.  **Native Bridge Fragility and Maintenance:** The most critical challenge is the maintenance of the native bridge. RN wrappers for on-device LLMs work by 'bridging' calls from JavaScript to a native library (e.g., a C++ engine like `llama.cpp` or `LiteRT-LM` packaged as an `.xcframework`). These LLM engines are evolving rapidly. When the native engine's API or ABI changes, the wrapper's bridge code breaks, leading to runtime errors. This forces developers into a difficult position: either wait for the wrapper's maintainer to issue a patch, or fork the wrapper and manually update the native C++/Objective-C bridge code themselves. This was precisely the case with `react-native-litert-lm`, where changes in the `LiteRT-LM` C API rendered the wrapper unusable with newer model bundles.

2.  **ABI/C API Mismatches:** This is the direct technical consequence of the bridge fragility. A wrapper compiled against one version of a native library's headers will fail to link or run correctly with a different version of that library if the function signatures, data structures, or enums have changed. This is a common source of 'engine creation errors' and crashes.

3.  **Complex Build and Configuration:** Integrating these native dependencies can be complex. For example, the `llama.rn` wrapper requires React Native's New Architecture (Fabric) to be enabled, which adds a layer of setup complexity. Furthermore, developers must correctly configure the native iOS project (`Podfile`, build settings) and enable specific entitlements, such as the 'Increased Memory Limit' capability, to allow the app to allocate the gigabytes of RAM required by the model without being terminated by the OS.

4.  **Memory Management Overhead:** React Native itself consumes memory. Running a multi-gigabyte LLM on top of the RN runtime, JavaScript engine, and the app's own state puts significant pressure on the device's memory. The wrapper must be impeccably designed to manage memory on the native side to avoid leaks and ensure the model can be loaded and run without exceeding the strict memory limits imposed by iOS.

# Fine Tune Provenance Challenges

It is exceptionally difficult to definitively verify that a community 'fine-tune' of a model like Gemma 4 is the one actually running on a device, due to the standard workflow for preparing models for mobile deployment. This challenge of provenance is not about technical impossibility, but about a lack of embedded, verifiable metadata in the final on-device artifact.

The common workflow proceeds as follows:

1.  **Offline Adapter Merging:** Instead of loading a large base model and a separate, smaller LoRA/PEFT adapter on the memory-constrained mobile device, the community standard is to perform a one-time merge operation offline (on a desktop or server). The weights of the LoRA adapter are mathematically merged into the weights of the base model, resulting in a new, standalone model with the fine-tuned behavior baked in.

2.  **Quantization and Conversion:** This new, merged model is then quantized (e.g., converted from 16-bit floats to 4-bit integers) to drastically reduce its file size and memory footprint. Following quantization, it is converted into a mobile-friendly runtime format, such as GGUF (for `llama.cpp`), a `.mlpackage` (for CoreML), or `.litertlm`.

3.  **Stripping of Metadata:** During this merge-quantize-convert process, the provenance of the model is typically lost. The final artifact, for example a `model-Q4_K_M.gguf` file, is essentially a container of quantized tensors. It does not contain embedded metadata that specifies its lineage, such as 'this model was created by merging `unsloth/gemma-4-E2B-it` with the `Maelstrome/lora-wave-session-r32` adapter.'

As a result, verifying a fine-tune becomes a matter of trust. When a repository like `sevenuphome/gemma4-iphone-demo` states it uses `unsloth/gemma-4-E2B-it-GGUF`, one must trust the documentation on the `unsloth` Hugging Face page. There is no programmatic way to inspect the GGUF file on the device and confirm its origin. This makes it hard to distinguish a genuine community fine-tune from a stock instruction-tuned model or another fine-tune entirely, making the validation of 'success stories' reliant on the quality and trustworthiness of the accompanying documentation rather than the artifact itself.

## Step Title

Target the MLX/mlx-swift Native Path First

## Description

The recommended action is to temporarily set aside the React Native wrapper problem and first build a minimal, native Swift proof-of-concept (PoC) application. This PoC will validate that your specific fine-tuned model can run on a physical iPhone using Apple's MLX framework.

The steps are as follows:

2.  **Build a Native Swift App:** Create a new, simple Xcode project for a native iOS app. 
3.  **Integrate MLX and Gemma 4 Support:** Add `ml-explore/mlx-swift` and `yejingyang8963-byte/Swift-gemma4-core` as Swift Package Manager dependencies. The `Swift-gemma4-core` package is crucial as it provides the necessary sidecar registration and implementation for Gemma 4's specific architecture (Proportional RoPE, PLE, etc.) that is not in the main `mlx-swift` library.
4.  **Load and Run the Model:** Following the example code in the `Swift-gemma4-core` README, load your converted MLX model and implement a simple UI to send a prompt and display the generated text stream.

## Rationale

This native-first approach is recommended because the research overwhelmingly shows that the most reliable successes for running fine-tuned Gemma 4 on iPhones are in native Swift applications. The MLX framework is Apple's official, high-performance solution for its hardware. Crucially, community projects like `Swift-gemma4-core` have already solved the difficult, Gemma-4-specific engineering challenges for iOS, providing you with a clear and proven implementation to follow. This strategy de-risks the most complex part of your project (on-device inference) by isolating it from the documented instability and ABI mismatch issues of existing React Native wrappers like `react-native-litert-lm`.
