Find publicly documented success stories of deploying a **fine-tuned Gemma 4 E2B or E4B model on a physical iPhone via some flavor of LiteRT**. The deployment runtime **must be LiteRT-based** — this is non-negotiable, because the target competition requires it. MLX, ExecuTorch, Core ML, `llama.cpp`/GGUF, and Web-LLM are **out of scope** for this query no matter how mature they are.

## What counts as "LiteRT-based"

Any of the following stacks qualify:

- **LiteRT-LM** (Google's higher-level LLM runtime, `.litertlm` container — magic bytes `LITERTLM`). Includes the C++ engine in `google-ai-edge/LiteRT-LM`, its iOS XCFramework, and any wrapper sitting on top.
- **MediaPipe LLM Inference** (`.task` bundle with inline sentencepiece tokenizer). MediaPipe LLM Inference is officially layered on top of LiteRT, so it counts even though its bundle format differs.
- **LiteRT core** (the TFLite successor, `TFL3` magic, `.tflite` files) running a Gemma 4 graph directly via `LiteRTInterpreter` / `tflite_flutter` / `react-native-fast-tflite` / similar.
- **Custom Swift app** linking the `LiteRT-LM` or `LiteRT` iOS XCFramework directly (no RN wrapper).
- **Forks** of `react-native-litert-lm` or `expo-llm-mediapipe` that have been patched against newer upstream.
- **`ai-edge-torch` / `litert-torch`** conversion pipelines whose output is then loaded by any of the above on iPhone.

## Context

- We have a LoRA fine-tune of `unsloth/gemma-4-E2B-it` merged via PEFT.
- Target: iOS (iPhone 17 Pro, iOS 26.4.2) via React Native (Expo SDK 54) with Metal acceleration. A custom Swift native module is acceptable if it lets us reach a working LiteRT-based path; pure-Swift apps are useful as references.
- Fine-tune adapter: `huggingface.co/Maelstrome/lora-wave-session-r32`.
- Current `.litertlm` export: `huggingface.co/Maelstrome/lora-wave-session-r32/litert-lm-v3/model.litertlm` (2.56 GB), built via `litert-torch-nightly 0.10.0.dev20260515`.
- Reference stock bundle that DOES load via `react-native-litert-lm@0.3.6` on the same device: `litert-community/gemma-4-E2B-it-litert-lm` (HF discussions: https://huggingface.co/litert-community/gemma-4-E2B-it-litert-lm/discussions — please mine these for working community recipes).

## What we have already ruled out — DO NOT re-recommend these paths

1. **`react-native-litert-lm@0.3.6` (Nitro)** with our `litert-torch`-built `.litertlm` — rejects every bundle variant we built with an engine creation error. Stock `litert-community/gemma-4-E2B-it.litertlm` loads fine on the same wrapper, but our fine-tune does not. Upstream issue `hung-yueh/react-native-litert-lm#9` is open and dead. Wrapper's bundled iOS XCFramework predates LiteRT-LM v0.11.0 (which introduced explicit Gemma 4 MTP support).
2. **Rebuilding the iOS XCFramework against `google-ai-edge/LiteRT-LM` `main`** and dropping into the wrapper's `node_modules` — diagnosed as an ABI mismatch: the wrapper's `HybridLiteRTLM.cpp` was built against an older `c/engine.h`; the rebuilt main framework changed the C API in 682 lines (sampler enum renames, conversation config now create+setters instead of single constructor, new tokenization APIs). Framework-only swap is insufficient.
3. **`expo-llm-mediapipe@0.6.0` / MediaPipe LLM Inference iOS SDK** with our LITERTLM container — rejects with `"failed to initialize engine: invalid argument sentencepiece tokenizer is not found in the model"`. (This rules out our specific LITERTLM bundle through this SDK, NOT MediaPipe-flavored `.task` bundles in general — a working Gemma 4 fine-tune `.task` recipe is still in scope.)
4. **MediaPipe Model Maker `.litertlm` output** (the original 5 GB bundle) — same wrapper rejection as #1. Same trap as the web-side blocker in `google-ai-edge/LiteRT-LM#2150`.
5. **`@mediapipe/tasks-genai`** in the browser — only registers a `TFL3` magic matcher; ignores `LITERTLM` containers. Google staff confirmed publicly there is no Gemma 4 fine-tune → web-`.task` conversion path. (Out of scope anyway — browser, not iPhone-native.)
6. **`litert-torch` Gemma 4 metadata builder** still has a literal `TODO(weiyiw)` in `model_ext/metadata_builder.py` falling back to Gemma 3's builder. Even users on `litert-torch#998`, `litert-torch#994` (Gemma 3n) report exports succeed but bundles either reject or generate pad tokens. **However**: if anyone has a working fork or patch of this builder, that IS in scope.
7. **Non-LiteRT runtimes** (MLX / mlx-swift, ExecuTorch / .pte, Core ML / .mlpackage, llama.cpp / GGUF / llama.rn, Web-LLM, wllama, cactus-llm-rn) — out of scope regardless of how well they work. **Do not** re-list these as recommended paths.

## What I'm specifically looking for

Concrete success stories — links to GitHub repos, blog posts, dev.to / Medium / Substack articles, hackathon writeups, Reddit threads, Discord/Slack screenshots, Hugging Face Discussions, or open-source apps — where someone has actually shipped a **fine-tuned Gemma 4** on a **physical iPhone** via a LiteRT-based stack. Especially:

1. **Forks or patches of `react-native-litert-lm`** that have been rebuilt against `LiteRT-LM` ≥ v0.11.0 (the version that added Gemma 4 MTP) and that load custom-built `.litertlm` bundles. Any GitHub fork, gist, or PR is interesting.
2. **Custom iOS Swift apps** linking the `LiteRT-LM` iOS XCFramework directly. The `mylovelycodes/LiteRTLM-Swift` repo demos stock Gemma 4 on iPhone 13 Pro — has anyone used it (or any analogous Swift app) with a **custom community fine-tune**? Are there any blog posts walking through the XCFramework integration?
3. **Working `litert-torch` / `ai-edge-torch` conversion recipes** that produce a `.litertlm` bundle which any current LiteRT-LM iOS runtime actually accepts for a fine-tuned Gemma 4. Look on HF Discussions on `litert-community/gemma-4-E2B-it-litert-lm`, `google-ai-edge/LiteRT-LM` issues/discussions, `ai-edge-torch` issues, and the Google AI Edge Discord.
4. **MediaPipe LLM Inference `.task` bundles** built from a **fine-tuned Gemma 4** (via MediaPipe Model Maker or any community tool) that actually load on iOS through `MediaPipeTasksGenAI` or `expo-llm-mediapipe`. Note: MediaPipe LLM Inference sits on LiteRT, so this still satisfies the constraint. The conversion error we hit was specifically about a LITERTLM container missing the sentencepiece tokenizer — a true `.task` build with inline tokenizer would be a different story.
5. **`LiteRT-LM` from-source iOS XCFramework builds** — anyone who has actually built `google-ai-edge/LiteRT-LM` from source against `main` (not relying on release tarballs missing Bazel runfiles) and shipped it on iPhone. Bazel + macOS workflow recipes are valuable.
6. **Google AI Edge Gallery iOS** or any Google-published reference iOS app loading Gemma 4 via LiteRT-LM with custom weights.
7. **Direct LiteRT (TFL3) deployment of Gemma 4** — has anyone exported Gemma 4 as raw `.tflite` (not LITERTLM-wrapped) and run inference through `LiteRTInterpreter` / `react-native-fast-tflite` on iPhone? This bypasses the LITERTLM container entirely while still being LiteRT.
8. **Forks of `litert-torch`'s `metadata_builder.py`** that have completed the `TODO(weiyiw)` Gemma 4 path, or any community patch.
9. **HF Discussions on `litert-community/gemma-4-E2B-it-litert-lm`** specifically — what conversion recipes, bundle layouts, or runtime version pairings has the community settled on for getting custom Gemma 4 weights into a loadable LITERTLM? Mine those discussions for working recipes.
10. **Anyone on Twitter/X, LinkedIn, YouTube, or hackathon writeups** demoing a fine-tuned Gemma 4 on iPhone specifically through LiteRT-LM, MediaPipe LLM Inference, or LiteRT core.

For each result, prefer:
- A direct GitHub repo / HF Space / HF Discussion URL
- Date of last update
- Specific evidence the model is **fine-tuned** (not stock `gemma-4-E2B-it`) and runs on a **physical iPhone** (not simulator/Mac/Android)
- The exact LiteRT flavor used (LiteRT-LM, MediaPipe LLM Inference, raw LiteRT/TFL3) and the conversion path
- The runtime version pairing (e.g., `litert-torch 0.X` + `LiteRT-LM v0.X` + wrapper `vX.X`) — runtime/builder version skew is the dominant failure mode here, so version data is critical

Negative results are useful too — if exhaustive search confirms nobody has publicly shipped a fine-tuned Gemma 4 on iPhone through any LiteRT-based path, say so, and identify which specific LiteRT sub-path is closest to viable (e.g., "from-source LiteRT-LM XCFramework + custom Swift app is the only theoretically open route, but no public examples exist").
