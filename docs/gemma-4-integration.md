# Gemma 4 + LoRA-Stack Integration Plan for WAVE (Gemma Hackathon MVP)

> **Scope:** a single shippable MVP for the Gemma hackathon. **Gemma 4 does
> every inference step. No Claude. No cloud LLM. No server.**
>
> **Architecture:** the app ships with **one Gemma 4 E2B-it base** plus a
> **stack of small LoRA adapters — one per clinical situation.** At runtime,
> an adapter manager hot-swaps the right LoRA onto the base model for each
> phase of the session. That turns WAVE into a set of specialized "models"
> (medication-acknowledgment-model, wave-peak-model, reflection-model, …)
> that all share one set of Gemma weights in memory.
>
> **Why this matters for the hackathon.** It's the Gemma pitch done right:
> one base model, many specialized behaviors, every byte trained by us on
> our own synthetic dataset, all running on-device.
>
> **Sources:**
> [Gemma 4 (DeepMind)](https://deepmind.google/models/gemma/gemma-4/) ·
> [ai.google.dev Gemma docs](https://ai.google.dev/gemma/docs) ·
> [Hugging Face: Welcome Gemma 4](https://huggingface.co/blog/gemma4)

---

## 1. MVP in one sentence

A judge opens WAVE, the browser (or phone) downloads **one Gemma 4 E2B-it
checkpoint and a bundle of small LoRA adapters** once, and every word of the
urge-surfing session — medication acknowledgment, body scan, wave rise / peak /
fall narration, reflection, notifications, insights — is generated by the same
Gemma base with a **different LoRA swapped in per phase**, fully on-device,
with the DevTools Network tab empty after first load.

## 2. The architectural pitch

```
                 ┌────────────────────────────────────────────┐
                 │ Gemma 4 E2B-it (INT4) — one base, in memory │
                 └────────────────────────────────────────────┘
                                       │
                ┌──────────────────────┼───────────────────────┐
                │                      │                       │
      ┌─────────┴────────┐   ┌─────────┴────────┐    ┌─────────┴────────┐
      │ LoRA: med-ack    │   │ LoRA: wave-peak  │ …  │ LoRA: reflection │
      │ ~25–60 MB, rank  │   │ ~25–60 MB        │    │ ~25–60 MB        │
      │ 8–16             │   │                  │    │                  │
      └──────────────────┘   └──────────────────┘    └──────────────────┘
              ▲                        ▲                       ▲
              │                        │                       │
        hot-swapped in by the Adapter Manager per session phase
```

- **One** Gemma 4 E2B base, shared across every surface, loaded once per tab /
  app launch.
- **Many** tiny LoRAs, each fine-tuned on its own synthetic dataset for one
  clinical situation (see §5). Crisis triage is the only surface that runs on
  **base Gemma, no LoRA**, because rule-based hotline routing must never drift
  from a fine-tune.
- **One** synthetic data pipeline — **Synthetix** — that, given a stack array
  of clinical conditions, generates a per-LoRA training set with a
  clinician-review gate (see §6).

This is what the user experiences: the app downloads the base weights **once**,
and a small set of LoRAs. As they use WAVE, the right LoRA is loaded on demand
for the current phase.

---

## 3. Base model + runtime (one choice)

**Base model:** **Gemma 4 E2B-it**, INT4-quantized. E2B (2.3 B effective /
5.1 B w/ embeddings, 128 k context, text + image + audio) is the Gemma 4 size
designed for phone-class and browser-class runtimes
([HF blog](https://huggingface.co/blog/gemma4)).

**Runtime — web demo (what runs for the hackathon):**
**`@huggingface/transformers` (transformers.js) + WebGPU**, loading the
pre-built Gemma 4 E2B ONNX bundle in a Web Worker. LoRA adapters are loaded via
PEFT-style adapter weights packaged as ONNX tensors (see §7). Every inference
runs in the browser.

- First visit: ~1.5 GB of INT4 base shards + ~150–400 MB of LoRA adapters
  stream from the site's static host and are cached in IndexedDB.
- Subsequent visits and every session: **zero network calls for inference**.
- Adapter swap target: **< 250 ms** to activate a new LoRA between phases
  (adapter weights live in GPU memory; swap is a pointer flip, not a reload).

**Runtime — production mobile (same architecture, post-hackathon port):**
**Gemma 4 E2B via LiteRT**, with the LoRA adapters loaded through the LiteRT
adapter interface. The same `Adapter Manager` TypeScript contract (see §7)
backs both runtimes.

**Fallback (single, scripted).** If WebGPU is unavailable, model load fails, or
any surface fails Zod validation twice in a row, that surface falls through to
a scripted narration bank keyed by `(matType × medicationStatus × phase)`. No
cloud LLM is ever called.

**Explicit cuts:**

- No other Gemma 4 sizes (E4B, 26B-A4B, 31B). One base.
- No self-hosted server, llama.cpp server, Ollama, or vLLM.
- No per-user fine-tune. LoRAs are trained by us ahead of time, shipped
  inside the app bundle, and identical for every patient. (On-device
  personalization is a clear post-MVP extension — see §10.)

## 4. The LoRA stack

One LoRA per clinical situation. Each LoRA is trained on a dataset generated
by Synthetix (§6) that covers every medication × status × trigger cell of
`PRD.md > Medication-Aware Prompt Logic` relevant to that phase.

| # | LoRA name               | Phase covered                  | MVP? | Size target | Rank | Notes |
|---|-------------------------|--------------------------------|------|-------------|------|-------|
| 1 | `lora-med-ack`          | §8.1 Medication acknowledgment | ✅ MVP | ~40 MB | 16 | Trained across all MAT × status cells. Pharmacology-accurate, trauma-informed tone. |
| 2 | `lora-body-scan`        | §8.2 Body-scan narration       | ✅ MVP | ~25 MB | 8  | Shorter copy; tone-focused. |
| 3 | `lora-wave-rise`        | §8.3 Wave — rise phase         | ✅ MVP | ~25 MB | 8  | Most directive; "ride the build". |
| 4 | `lora-wave-peak`        | §8.3 Wave — peak phase         | ✅ MVP | ~30 MB | 8  | Most grounding; never celebrating. |
| 5 | `lora-wave-fall`        | §8.3 Wave — fall phase         | ✅ MVP | ~25 MB | 8  | Normalizing; can acknowledge the drop. |
| 6 | `lora-reflection`       | §8.4 Post-session reflection   | ✅ MVP | ~40 MB | 16 | Must weave in the patient's numeric drop and longitudinal context. |
| 7 | `lora-notification`     | §8.5 Prophylactic notifications | stretch | ~20 MB | 8 | Short; down-weighted tone when ignored repeatedly. |
| 8 | `lora-insights`         | §8.6 Weekly pattern insights   | stretch | ~50 MB | 16 | Runs on the full session history; longest context. |
| — | **(no LoRA — base only)** | §8.7 Crisis triage          | ✅ MVP | 0 MB  | —  | Base Gemma + rule-based routing. Never fine-tuned. |

**MVP target:** ship **LoRAs 1–6** trained end-to-end for the hackathon (the
full session path). LoRAs 7 and 8 are stretch for the weekend; their
Synthetix data specs are defined so the training pipeline can be run the
same way later.

**Why split this way**

- **Each phase has a distinct clinical job.** Wave-rise ≠ wave-peak ≠
  wave-fall in MBRP. Splitting LoRAs by phase lets each adapter concentrate
  its capacity on one narrow tone target.
- **Adapters are tiny.** A rank-8 LoRA on E2B is ~25 MB; a rank-16 is ~50 MB.
  All seven MVP adapters fit in under ~250 MB — a small fraction of the
  base weights.
- **Independent training and review.** Each LoRA has its own synthetic
  dataset, rubric, clinician review batch, and eval. A regression in
  `lora-wave-peak` can't break `lora-med-ack`.
- **Explicit safety boundary.** Crisis triage is the one place we
  specifically do **not** want a fine-tune. Keeping it on base Gemma makes
  that boundary visible in the adapter manifest.

## 5. Surfaces, inputs, outputs, and which LoRA runs

Each surface runs through the same wrapper:

```ts
generateJSON<T>(loraId: LoRAId | null, schema: ZodSchema<T>, system: string, user: string): Promise<T>
```

Passing `null` means "run on base Gemma only" — used for Crisis triage.
Everything else names its dedicated LoRA.

### 5.1 Medication-aware acknowledgment — `lora-med-ack`

**Inputs**

```ts
type AckInput = {
  intensity: number;            // 1..10
  medicationStatus: "on_time" | "late" | "missed" | "none";
  matType: "buprenorphine" | "naltrexone" | "methadone" | "vivitrol" | "none";
  trigger: "social" | "stress" | "physical" | "unknown" | "other";
  hoursSinceDose?: number;      // 0..48
  localeTimeOfDay: "morning" | "midday" | "evening" | "late_night";
};
```

**Output contract**

```ts
type AckOutput = {
  acknowledgment: string;        // <= 280 chars, 2–3 sentences
  pharmacologyClaim: {
    medication: AckInput["matType"];
    claim: string;               // short factual statement
    citation: "FDA_LABEL" | "SAMHSA_TIP63" | "MBRP_FACILITATOR" | "NONE";
  };
  crisisSignalDetected: boolean;
  nextPhase: "body_scan";
};
```

**Decoding:** T = 0.3, max 180 tokens, JSON-only.
**p95 latency:** 2.5 s on WebGPU laptop.
**Safety rails:** pharmacology allow-list (reject `{increase, decrease, stop, start, double, skip}` + `{dose, medication}`), crisis keyword pre-filter on any attached journal.
**Fallback:** scripted line per `(matType, medicationStatus)` cell from the PRD matrix.

### 5.2 Body-scan narration — `lora-body-scan`

**Inputs**

```ts
type BodyScanInput = AckInput & {
  bodyLocation: "chest" | "jaw" | "shoulders" | "legs" | "stomach" | "other";
};
```

**Output**

```ts
type BodyScanOutput = {
  narration: string;                 // <= 320 chars
  breathCount: 3 | 4 | 5;
  sensationLabel: "tight" | "warm" | "cold" | "fluttery" | "heavy" | "absent";
};
```

**p95 latency:** 1.5 s.
**Safety rails:** must not name any substance (lexical post-filter → fallback).
**Fallback:** 18 scripted body-scan lines keyed by `bodyLocation`.

### 5.3 Adaptive wave narration — `lora-wave-rise` / `lora-wave-peak` / `lora-wave-fall`

Three separate adapters, one per phase. The Adapter Manager swaps LoRAs at
each phase transition (§7). In-between slider-driven micro-nudges stay
rule-based; the LoRA is only invoked at the transition.

**Inputs** (one call per transition)

```ts
type WaveInput = AckInput & {
  phase: "rise" | "peak" | "fall";
  currentIntensity: number;
  intensityTrendLast60s: "up" | "flat" | "down";
  elapsedSeconds: number;           // 0..480
};
```

**Output**

```ts
type WaveOutput = {
  narration: string;                // <= 220 chars
  pacingHint: "slower" | "hold" | "faster";
  encouragement: "grounding" | "normalizing" | "celebrating";
};
```

**p95 latency:** 1.8 s per transition, streamed. **Total adapter swap overhead per session: three swaps × ~250 ms = ~0.75 s budgeted out of the 5–8 min session.**
**Per-phase safety:**

- `rise`: directive, never shaming.
- `peak`: grounding or normalizing, **never celebrating** (hard Zod constraint on `encouragement`).
- `fall`: can acknowledge the drop, celebrating allowed only here.

**Fallback:** 3 scripted lines per phase by `(matType, trigger)`.

### 5.4 Post-session reflection — `lora-reflection`

**Inputs**

```ts
type ReflectionInput = {
  intakeIntensity: number;
  endingIntensity: number;
  durationSeconds: number;
  medicationStatus: AckInput["medicationStatus"];
  matType: AckInput["matType"];
  sessionsCount: number;
  optionalJournalText?: string;     // <= 500 chars
};
```

**Output**

```ts
type ReflectionOutput = {
  insightOneLine: string;
  journalPromptQuestion: string;
  suggestedNextStep: "call" | "walk" | "water" | "hands" | "rest";
  crisisSignalDetected: boolean;
};
```

**p95 latency:** 2.0 s.
**Safety rails:** `insightOneLine` must contain the numeric ending intensity. Journal text that trips the crisis filter routes to §5.7.

### 5.5 Prophylactic notifications — `lora-notification` *(stretch)*

**Inputs**

```ts
type NotificationInput = {
  predictedWindow: { weekday: number; startHour: number; endHour: number; relativeRisk: number };
  minutesUntilWindow: 15;
  recentDropDelta?: number;
  ignoredWindowCount: number;
};
```

**Output**

```ts
type NotificationOutput = {
  title: string;                // <= 40 chars
  body: string;                 // <= 120 chars
  cta: "Open WAVE" | "Log medication" | "Snooze";
};
```

**p95 latency:** 1.0 s.
**Safety rails:** never name a substance. If `ignoredWindowCount >= 3`, title is forced to `"Here when you want us"` and only `body` comes from the LoRA.

### 5.6 Weekly pattern insights — `lora-insights` *(stretch)*

**Inputs**

```ts
type InsightsInput = {
  sessionHistory: Array<{
    startedAt: string;
    intakeIntensity: number;
    endingIntensity: number;
    medicationStatus: AckInput["medicationStatus"];
    trigger: AckInput["trigger"];
    bodyLocation?: BodyScanInput["bodyLocation"];
  }>;
  riskWindowModel: {
    windows: Array<{ weekday: 0|1|2|3|4|5|6; startHour: number; endHour: number; relativeRisk: number }>;
    medicationCorrelation: number;
  };
};
```

**Output**

```ts
type InsightsOutput = {
  patterns: Array<{
    kind: "time_of_day" | "trigger_frequency" | "medication_correlation" | "body_location";
    plainEnglish: string;
    confidence: "low" | "medium" | "high";
  }>;
  actionableSuggestion: {
    text: string;
    targetWindow?: { weekday: number; startHour: number; endHour: number };
  };
  crisisSignalDetected: false;
};
```

**p95 latency:** 8 s. Runs off the session critical path (weekly background).

### 5.7 Crisis triage — **base Gemma, no LoRA**

Routing is **rule-based**; Gemma only renders the copy.

Rule:
- Suicidality → `routeTo="988"`, `continueSession=false`.
- "Used already" + lethal-dose markers → `routeTo="local_emergency"`, `continueSession=false`.
- Otherwise → `routeTo="samhsa_helpline"`, `continueSession=true`.

The LoRA stack is explicitly **bypassed** for this surface — the Adapter Manager is called with `loraId=null`. This is enforced in type system by making `CrisisInput` the only input shape whose handler signature takes `loraId: null`.

**Safety rail:** `copy` is post-validated to contain the hotline number that matches `routeTo`. Mismatch → hard-coded canned copy.

Matches `AGENTS.md > Domain Constraints > Crisis handoff`.

## 6. Synthetix — the per-LoRA synthetic-data pipeline

One round of fine-tuning per LoRA. One loop to build the dataset. No batched
review, no held-out splits, no A/B rubric scoring. The loop is four steps:

1. **Seed with a small human-written set.**
2. **Expand with a Gemma model** into `N` synthetic examples.
3. **Spot-check those synthetics in a small UI.** Leave feedback on anything
   wrong.
4. **Regenerate** using that feedback. Repeat (3)-(4) until a spot-check
   passes with **no errors**. Then fine-tune, once.

### 6.1 Step 1 — human seed set

For each LoRA, a clinician (or a domain-informed developer) writes a small
set of hand-crafted (structured input → JSON output) examples matching the
Zod contract in §5. Seed sizes are small on purpose:

| LoRA                 | Human seed examples |
|----------------------|---------------------|
| `lora-med-ack`       | 40                  |
| `lora-body-scan`     | 20                  |
| `lora-wave-rise`     | 20                  |
| `lora-wave-peak`     | 25                  |
| `lora-wave-fall`     | 20                  |
| `lora-reflection`    | 25                  |
| `lora-notification`  | 15 *(stretch)*      |
| `lora-insights`      | 20 *(stretch)*      |

Seeds live as typed TypeScript data under `clients/synthetix/seeds/<lora-id>.ts`
so they are reviewable in code review.

The seed set is **not** a full coverage of the stack array (§5) — it is the
variety seed that teaches the generator what "good" looks like for that
LoRA's surface. The generator handles the combinatorial coverage.

### 6.2 Step 2 — Gemma-generated synthetics

A Gemma model running on the developer workstation (**Gemma 4 31B-it** if
available, otherwise **26B-A4B-it**; falls back to E4B-it on smaller dev
machines) expands the seed set into `N` synthetic examples per LoRA.

| LoRA                 | Synthetic examples (`N`) |
|----------------------|--------------------------|
| `lora-med-ack`       | 400                      |
| `lora-body-scan`     | 200                      |
| `lora-wave-rise`     | 200                      |
| `lora-wave-peak`     | 250                      |
| `lora-wave-fall`     | 200                      |
| `lora-reflection`    | 250                      |
| `lora-notification`  | 120 *(stretch)*          |
| `lora-insights`      | 150 *(stretch)*          |

Total MVP (LoRAs 1–6): **~1 500 synthetic examples** across the stack.
These are the only examples that go into training — we do not reserve a
held-out split. The spot-check in §6.3 replaces a held-out eval.

The generator is prompted with: (a) the surface's Zod schema, (b) the human
seed set, (c) a structured input sampled from the §5 stack axes, (d) the
tone rules from `AGENTS.md > Domain Constraints` (no toxic positivity, no
substance naming, no pharmacology directives, crisis-handoff rules), and
(e) "JSON only, match this schema". Temperature 0.7 for variety.

> The generator model never ships to users. It runs only in the dev
> pipeline. The user-facing model is 100 % Gemma 4 E2B-it + our LoRAs.

### 6.3 Step 3 — spot-check UI with feedback

A local Next.js page under `clients/app/synthetix-review/` shows the
clinician a sampled subset of the generated synthetics. One example at a
time, showing the structured input and the generated JSON output, with three
controls:

- **Looks good.** Marks this example as clean.
- **Has a problem.** Opens a free-text feedback field. The clinician
  describes what's wrong ("uses the word 'alcohol'", "says the patient
  should increase their dose", "tone feels shaming", "encouragement is
  'celebrating' during the peak phase", etc.).
- **Skip / unsure.** Next example.

Spot-check sample size is small: **~30 examples per LoRA per round**, drawn
uniformly across the stack cells that were generated. Spot-check results are
logged to `clients/synthetix/runs/<lora-id>/<run-id>/spotcheck.json`.

A round **passes clean** when the clinician marks all sampled examples
"Looks good" — i.e. zero problems flagged in the sampled set.

If any example is flagged, the round fails and we go to §6.4.

### 6.4 Step 4 — regenerate with feedback

When a spot-check round has problems flagged, all feedback strings for that
round are aggregated and fed back into the generator prompt as an explicit
"Previous issues to avoid" block. The generator re-runs and produces a new
batch of `N` examples for the LoRA.

Return to §6.3 and spot-check the new batch.

Repeat until a round passes clean — typically 1–3 iterations per LoRA.

Each round (seed → generate → spot-check → feedback) is stored under
`clients/synthetix/runs/<lora-id>/<run-id>/` with the generator prompt, the
generated examples, and the spot-check result, so the full provenance of the
final dataset is reviewable.

### 6.5 Step 5 — fine-tune, once

Once a LoRA's spot-check passes clean, we run **one** QLoRA training job on
the last-approved generated dataset (the full ~400 examples for
`lora-med-ack`, ~200 for `lora-body-scan`, etc.). No subsequent rounds, no
iterative fine-tuning, no RLHF. The resulting adapter is the shipped
adapter; it goes into the manifest (§7.3) with the run ID that produced its
dataset.

Training details live in §8. A LoRA's training run is gated on:

- Spot-check `pass: true` in the latest run file.
- Clinician initials on the spot-check (same person who approved it).
- All synthetic examples in the run parse against the Zod schema.

There is no held-out test or delta-vs-base bar. The spot-check is the eval —
it is small, human, and domain-expert-driven, which is what matters for
clinical tone. If a shipped LoRA underperforms base at demo time, we revert
the manifest entry and that surface runs on base + prompt.

## 7. Adapter Manager — how LoRAs are swapped at runtime

### 7.1 Contract (same on browser and mobile)

```ts
type LoRAId =
  | "lora-med-ack"
  | "lora-body-scan"
  | "lora-wave-rise"
  | "lora-wave-peak"
  | "lora-wave-fall"
  | "lora-reflection"
  | "lora-notification"
  | "lora-insights";

interface AdapterManager {
  /** Ensures the base model is loaded (once per process). */
  ensureBaseLoaded(): Promise<void>;

  /** Ensures the adapter is resident in GPU memory. No-op if already loaded. */
  preload(loraId: LoRAId): Promise<void>;

  /** Runs generation on base + the given LoRA (null = base only). */
  generate<T>(
    loraId: LoRAId | null,
    schema: ZodSchema<T>,
    system: string,
    user: string,
    opts?: { temperature?: number; maxTokens?: number },
  ): Promise<T>;

  /** Adapter manifest + hash pinning for cache invalidation. */
  manifest(): AdapterManifest;
}
```

### 7.2 Lifecycle during a session

```
  t = 0        App open → ensureBaseLoaded()   (one-time per tab)
               Intake UI shows while base streams in; opportunistic
               preload() for lora-med-ack in parallel.

  t = intake   preload("lora-med-ack")  (no-op if already cached)
               generate("lora-med-ack", AckOutputSchema, …)

  t = body     preload("lora-body-scan") kicked off during the
               acknowledgment-to-body-scan transition (UI has ~1 s here)
               generate("lora-body-scan", BodyScanOutputSchema, …)

  t = wave↑    preload("lora-wave-rise"); generate(...)
  t = wave●    preload("lora-wave-peak");  generate(...)
  t = wave↓    preload("lora-wave-fall");  generate(...)

  t = reflect  preload("lora-reflection"); generate(...)

  t = crisis?  generate(null, CrisisOutputSchema, …)    ← base only, no LoRA
```

Key invariants:

- **Only one LoRA is active at a time.** Swap is a pointer flip in GPU
  memory, not a reload. Target < 250 ms.
- **Adapters preload just-in-time.** The transition animations (intake →
  body scan, etc.) are used as cover for the preload, so the swap is
  invisible to the patient.
- **LRU eviction.** If adapter memory pressure is high on low-end devices,
  evict the least-recently-used LoRA. Base is pinned and never evicted.

### 7.3 Packaging & versioning

The app ships an **adapter manifest** alongside the base weights:

```ts
// clients/lib/gemma/adapter-manifest.ts
type AdapterManifest = {
  baseModel: {
    id: "google/gemma-4-E2B-it";
    quantization: "int4";
    sha256: "…";
  };
  adapters: Array<{
    loraId: LoRAId;
    version: string;                  // semver, e.g. "1.0.3"
    sha256: string;                   // content hash for cache invalidation
    url: string;                      // same-origin static asset
    sizeBytes: number;
    synthetixRunId: string;           // points at clients/synthetix/runs/<lora-id>/<run-id>/
    spotCheckPassed: true;            // literal true; false never ships
    approvedBy: string;               // clinician initials on the clean spot-check
  }>;
};
```

Properties:

- Manifest is fetched once and cached. If a LoRA's `sha256` changes in a
  later release, only that adapter re-downloads — base stays cached.
- Manifest is signed (app-level signature) so a rogue update can't silently
  swap a safety-critical adapter.
- Bundling strategy for first install: ship LoRAs 1–6 **inside the initial
  bundle** (~200 MB total) so the full session path is available
  immediately; ship LoRAs 7–8 lazily on first use.

### 7.4 Fine-tune portability between runtimes

- **Training** happens on the HF `transformers` + PEFT stack.
- **Web runtime:** adapters are converted to ONNX tensor packs that
  transformers.js + WebGPU can apply at inference time.
- **Mobile runtime:** adapters are converted to LiteRT's adapter format.
- The conversion is automated in `clients/synthetix/export/`; CI fails if a
  LoRA's web and mobile builds diverge in eval score by > 2 %.

## 8. Training recipe (per LoRA)

One training run per LoRA. No held-out split. No iterative fine-tune. The
spot-check in §6.3 is the eval.

- **Framework:** [Unsloth](https://github.com/unslothai/unsloth) + TRL +
  QLoRA, targeting `google/gemma-4-E2B-it`.
- **Adapter shape:** LoRA on `q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj`.
- **Rank / alpha / dropout:** rank 8 or 16 per §4; alpha = 2 × rank;
  dropout 0.05.
- **Optimization:** AdamW, lr = 2e-4 with cosine schedule, warmup 3 %,
  batch size 8 (gradient accumulation as needed), 3 epochs, save the final
  checkpoint.
- **Dataset:** the full set of generated examples from the spot-check-clean
  Synthetix run (§6.2 sizes). No split.
- **Compute:** fits comfortably on a single A100 40 GB; each LoRA trains in
  10–30 minutes on the §6.2 dataset sizes.

No RLHF. No multi-stage fine-tune. One round of QLoRA on spot-check-clean
synthetic data is the shipped recipe.

## 10. Post-MVP (explicit, not in hackathon scope)

- **`lora-notification` and `lora-insights`** trained and shipped
  (architecture already supports them; it's just data + compute).
- **Per-user personalization.** An 8-rank user adapter trained **on-device**
  from the patient's own session history (journal text, which triggers
  respond to which lines, etc.), stacked on top of the shipped LoRAs. LoRA
  composition is already supported by PEFT; privacy stays intact because
  training never leaves the device.
- **Multimodal LoRAs.** Once the photo and voice intake features ship
  (`PRD.md > Out of Scope`), each gets its own LoRA on Gemma 4 E2B's
  vision/audio tower, trained the same way.
- **Clinician-reviewed continuous refresh.** Monthly Synthetix runs on new
  seed material (clinician field notes, updated SAMHSA TIPs) produce new
  LoRA releases. Only the adapter bytes re-download — base weights never
  change.

## 12. Demo script (what the judges see)

1. Open WAVE in Chrome/Edge with WebGPU. A "Loading WAVE's models" screen
   streams **one Gemma 4 E2B base + six small LoRAs** in parallel with a
   progress bar. This happens once.
2. Judge opens DevTools → Network → toggles **Offline**.
3. Judge taps **7/10**, **"took Suboxone on time"**, **stress**.
4. The session fires: `lora-med-ack` for the acknowledgment, `lora-body-scan`
   for the body scan, `lora-wave-rise` → `-peak` → `-fall` across the wave,
   `lora-reflection` for the closing line. A "now active: lora-wave-peak"
   indicator in the DevTools UI (dev-mode only) shows the swap happening.
5. Judge sees **no network requests** for any inference.
6. Pitch: *"One Gemma 4 base. Seven LoRA adapters, each fine-tuned on its
   own synthetic clinical dataset. The user downloads it once; everything
   runs on-device forever after. This is the shape we think good
   on-device AI takes — a small shared base, and a stack of specialized
   behaviors you can audit one at a time."*

## 13. Honest risks

- **Hackathon scope of 6 trained LoRAs.** With the simplified Synthetix
  loop (seed → generate → spot-check → regenerate → one training run), each
  LoRA's full cycle is a few hours of generator + spot-check time plus
  30–90 minutes of QLoRA on a single A100. Mitigation: LoRAs 1, 4, and 6
  (med-ack, wave-peak, reflection) are the demo-critical ones; if we run
  out of time the other three fall back to base + prompt.
- **WebGPU LoRA support.** transformers.js supports adapter loading, but
  the tooling is younger than LiteRT's. Mitigation: prove the swap works
  for a single LoRA end-to-end before scaling to the full stack; fall back
  to pre-compiled "base + adapter merged" ONNX files per LoRA if dynamic
  swapping is too slow (trades memory for latency).
- **Synthetix quality.** Synthetic data generated by a larger Gemma can be
  subtly hallucinatory on pharmacology. Mitigation: the automatic rubric
  filter (§6.3) is tight, and the clinician review gate is mandatory for
  every batch. No unreviewed data enters a training set.
- **Crisis drift.** The whole reason crisis triage stays on **base, no
  LoRA**. If a future team considers adding `lora-crisis`, this doc is the
  red flag — don't.
- **Adapter sprawl.** Eight LoRAs is already a lot to keep reviewed. The
  manifest (§7.3) makes ownership and version explicit; adding a ninth
  requires a PR that updates the manifest schema and adds an entry to
  `PRD.md`.

## 14. Alignment with `PRD.md` and `AGENTS.md`

- `PRD.md > Medication-Aware Prompt Logic` is the canonical matrix every
  Synthetix stack is checked against. A LoRA that cannot reproduce its row
  of the matrix on held-out test data does not ship.
- `AGENTS.md > Domain Constraints > Crisis handoff` is the reason crisis
  triage is the one surface without a LoRA.
- `AGENTS.md > Domain Constraints > Offline-first` is satisfied: everything
  — base + adapters + fallbacks — is in IndexedDB / the app bundle after
  first load.
- `AGENTS.md > Code Style > Clinical copy lives in data, not in components`
  is satisfied by both the prompt templates and the LoRAs themselves: the
  model's learned behavior is serialized as a small, versioned,
  clinician-approved artifact (§7.3).
