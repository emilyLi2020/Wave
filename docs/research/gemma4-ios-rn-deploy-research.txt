Deep Research Prompt: Deploying a fine-tuned Gemma 4 E2B on iPhone via React Native
====================================================================================

BACKGROUND
----------
I'm shipping a fine-tuned `unsloth/gemma-4-E2B-it` LoRA (merged + quantized
to GGUF Q4_K_M, ~3 GB) on-device for an iPhone meditation/recovery app
(WAVE). Stack:

  - React Native via Expo (managed/dev client workflow, EAS Build for
    custom native modules)
  - iOS target: iPhone 15 / 16 (A17 Pro / A18 / A18 Pro), iOS 18+
  - Existing on-device ML in the app: Silero VAD via onnxruntime-react-native,
    Kokoro TTS via sherpa-onnx. So we already have RN-bridged native
    audio/ML modules working in production.
  - Browser inference (web build of the app, sibling target) ships
    Gemma via wllama (llama.cpp WebAssembly) and parses tool calls from
    plain-text output. So we have a known-working wllama setup we can
    point a native iOS runtime at.
  - Prize requirement mandates the model be fine-tuned and run on-device.

WHAT WE'VE TRIED ON iOS
-----------------------
1. react-native-litert-lm@0.3.6 — the Google Pixel team's LiteRT-LM
   wrapper for RN. Stuck behind the upstream LiteRT-LM C++ release;
   their .litertlm bundle format has no browser consumer and the iOS
   side of the wrapper lags upstream. Dead end for our needs.

2. MediaPipe LLM Inference (Google AI Edge) — the official Google
   on-device LLM SDK. We pivoted to this for the prize requirement.
   Has iOS SDK. Consumes a .task bundle (LiteRT-LM under the hood).
   Unclear whether the React Native side has a maintained bridge or
   if we'd be writing the Obj-C/Swift bridge ourselves.

3. Considered but not yet tried:
   - llama.rn (https://github.com/mybigday/llama.rn) — llama.cpp ported
     to React Native, GGUF support, claimed iOS performance
   - cactus (https://github.com/cactus-compute/cactus) — newer RN
     LLM runtime
   - Direct Swift bridge to llama.cpp from RN
   - CoreML conversion (Gemma -> CoreML to hit Apple Neural Engine)

CONSTRAINTS / REQUIREMENTS
--------------------------
- Model file ~3 GB Q4_K_M. Needs runtime download + cache (we can't ship
  it inside the app bundle for App Store).
- App must work offline after first launch.
- Tool calling: model emits literal text `endConversation{cravingScore:N,
  obstacleCategory:CAT}\n<closing speech>`. We need to regex-parse the
  first line at the runtime layer.
- Streaming token output to UI (for a chat-style check-in).
- Background/foreground transitions: pause generation if backgrounded,
  resume on return.
- Expo workflow: the runtime must be installable as an Expo dev client
  module (or via prebuild) — pure CocoaPods/SPM works fine, but plugins
  that require ejecting from Expo entirely are a hard sell.
- Apple Silicon: A17 Pro / A18 / A18 Pro. 8 GB RAM. Apple Neural Engine
  available but rarely used by general LLM runtimes.
